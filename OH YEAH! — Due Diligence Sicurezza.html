SICUREZZA APPLICATIVA
  • ✓ Autenticazione: Argon2id m=65536, t=2, p=1 (OWASP raccomandato). Password policy: min. 10 car., maiuscola + minuscola + numero. Test contro dizionario (nome ristorante, email).
  • ✓ Session token: UUID4, stored in Supabase, TTL 30 giorni, invalidazione esplicita al logout. Cookie HttpOnly via extra-streamlit-components.
  • ✓ Rate limiting login: 5 tentativi / 15 minuti per IP+email. Implementato DB-backed (persistente across restart).
  • ✓ Rate limiting reset password: 1 richiesta / 5 minuti per email — DB-backed (migrazione 046).
  • ✓ File upload: validazione magic bytes (non solo estensione), limite 100 file / 200 MB per sessione. Formati accettati: .xml, .p7m, .zip.
  • ✓ Error handling: nessun stack trace esposto in UI. Eccezioni loggati server-side, messaggi utente generici.
  • ✓ Iniezione SQL: ZERO query raw — 100% Supabase SDK con parametri tipizzati.
  • ✓ XSS: output HTML tramite unsafe_allow_html solo per componenti interni; input utente non riflesso in HTML non sanificato.
  • ✓ Webhook (Supabase → App): HMAC-SHA256, anti-replay 5 minuti, whitelist CIDR Supabase, idempotenza tramite hash payload.
  • ✓ Segreti: zero credenziali nel codice sorgente. Audit git history pulito. secrets.toml e .env in .gitignore.

WEBHOOK E WORKER AUTOMATICO
  • ✓ Autenticità: ogni evento Supabase → App firmato con HMAC-SHA256. Firma verificata prima di qualsiasi elaborazione.
  • ✓ Anti-replay: timestamp nell'header, rifiuto richieste con delta > 5 minuti. Idempotenza tramite hash SHA256 del payload (duplicati ignorati).
  • ✓ SSRF protection: whitelist CIDR degli indirizzi Supabase. Richieste da IP non autorizzati rifiutate con HTTP 403.
  • ✓ GDPR & Pulizia dati: file XML delle fatture purgati automaticamente dopo il processing. Nessuna copia residua su disco.
  • ✓ Worker: GitHub Actions cron ogni 5 min. Retry esponenziale (3 tentativi). Fatture non elaborate → coda fatture_queue con stato tracciabile.

OWASP TOP 10 — STATO MITIGAZIONE
Vulnerabilità OWASP 2021 Stato Misure implementate
A01 — Broken Access Control ✓ Mitigato RLS PostgreSQL + filtro user_id+ristorante_id applicativo. Admin route con verifica ruolo server-side.
A02 — Cryptographic Failures ✓ Mitigato Argon2id per password, HMAC-SHA256 per webhook, TLS 1.3 in transito (Supabase/Streamlit), AES-256 a riposo (Supabase).
A03 — Injection (SQL, XSS) ✓ Mitigato 100% Supabase SDK parametrizzato. Input utente non riflesso in HTML non sanificato. Nessuna query concatenata.
A04 — Insecure Design ✓ Mitigato Multi-tenancy by design (ristorante_id obbligatorio in ogni tabella). Rate limiting DB-backed. Separazione ruoli admin/utente.
A05 — Security Misconfiguration ✓ Mitigato Zero credenziali nel codice. secrets.toml gitignored. ENV vars per tutti i segreti. Warning se ADMIN_EMAILS non configurata.
A06 — Vulnerable Components ✓ Mitigato requirements-lock.txt con hash pinned. Dependabot attivo. CVE cryptography + PyJWT risolti al 21/03/2026.
A07 — Auth Failures ✓ Mitigato Argon2id, rate limiting login+reset DB-backed, session UUID4 con TTL, invalidazione al logout, password policy.
A08 — Data Integrity Failures ✓ Mitigato HMAC-SHA256 su webhook, idempotenza, requirements-lock.txt con hash SHA256, pipeline CI bloccante.
A09 — Logging & Monitoring ✓ Mitigato Log strutturati rotating 50 MB×10, separati per modulo. Uptime check ogni 5 min con alert email. 172 test CI.
A10 — SSRF ✓ Mitigato Whitelist CIDR Supabase sul webhook. Nessuna fetch di URL arbitrari da input utente. OpenAI SDK (no URL custom).
